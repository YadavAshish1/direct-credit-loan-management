# DirectCredit - End-to-End Workflow & Data Architecture

This document explains the complete workflow of the DirectCredit Loan Management System, tracking how data flows between the user interfaces (Frontend & Admin), the Backend APIs, and the PostgreSQL database tables.

---

## 1. Database Schema & Data Flow Mapping

The system relies on 5 primary PostgreSQL tables. Here is how data moves between them:

1. **`users` Table**
   - **What it stores:** Authentication credentials (hashed passwords), basic details, and employment/financial profiles of both customers and admins.
   - **Data Origin:** Registration page (`/register`) and Profile Update page.
   - **Where it is used:** Used by the Auth system for login, and by the **Eligibility Engine** to verify age, employment stability, and income.

2. **`loan_applications` Table**
   - **What it stores:** The core loan request details (amount, tenure, loan type, status, and disbursement details).
   - **Data Origin:** The "Apply for Loan" form on the Customer portal.
   - **Where it is used:** Displayed on both the Customer's dashboard and the Admin's application list. Drives the entire state machine (`submitted` -> `under_review` -> `approved` -> `disbursed`).

3. **`credit_reports` Table**
   - **What it stores:** Simulated CIBIL/Experian bureau data (Credit Score, total accounts, outstanding debt, credit utilization).
   - **Data Origin:** Generated automatically by the `CreditBureauService` when an application is submitted or when a user checks their score.
   - **Where it is used:** Consumed by the **Eligibility Engine** to determine loan eligibility and calculate the Fixed Obligation to Income Ratio (FOIR).

4. **`eligibility_checks` Table**
   - **What it stores:** The final mathematical result of the loan assessment (Eligibility Score out of 100, boolean `is_eligible`, and a JSON array of which rules passed or failed).
   - **Data Origin:** Generated by the `EligibilityEngine` service in the backend immediately after an application is submitted.
   - **Where it is used:** Displayed on the Admin's "Review Application" page to help them make a decision.

5. **`audit_logs` Table**
   - **What it stores:** Immutable records of sensitive actions (e.g., "Loan Approved", "Loan Disbursed", "Eligibility Engine Ran").
   - **Data Origin:** Automatically inserted by backend controllers whenever a state changes.
   - **Where it is used:** Displayed in the Admin Portal under "System Audit Logs" for compliance and tracking.

## 2. The Complete Loan Application Journey

Here is the exact step-by-step workflow from the moment a user signs up to the moment money hits their bank account:

### Step 1: Customer Profiling
* **Action:** The customer registers and fills out their profile (PAN, Aadhaar, Income, Employment Type).
* **Data Flow:** Saved into the `users` table.

### Step 2: Loan Submission
* **Action:** The customer submits a loan application for ₹5,00,000 for 36 months.
* **Data Flow:** A new row is inserted into `loan_applications` with `status = 'submitted'`.

### Step 3: Triggering the Eligibility Pipeline (Automated)
* **Action:** The moment the loan is submitted, the backend automatically triggers the Assessment Pipeline.
* **Bureau Fetch:** The system queries `credit_reports` for the user. If none exists, the `CreditBureauService` generates one instantly (see Section 3 for the math).
* **Engine Evaluation:** The `EligibilityEngine` loads the `user`, `loan_application`, and `credit_report` data into memory. It runs 7 strict banking rules.
* **Data Flow:** The engine's result is saved into `eligibility_checks`. The `loan_application` status is updated to `eligible` or `ineligible` based on the engine's final verdict.

### Step 4: Admin Review
* **Action:** An Administrator logs into the Admin Portal and opens the application.
* **View:** The Admin sees a combined view of the `user` profile, the `loan_application` request, and the `eligibility_checks` rule-by-rule breakdown.
* **Decision:** The Admin clicks "Approve".
* **Data Flow:** The `loan_applications` status updates to `approved`. The `audit_logs` table records that this specific Admin approved the loan.

### Step 5: Loan Disbursement
* **Action:** The Admin enters NEFT/RTGS details and clicks "Process Transfer".
* **Data Flow:** The `loan_applications` table updates its `disbursement` JSON column with the transaction reference, and the status changes to `disbursed`. The customer sees the fireworks on their dashboard!

## 3. Deep Dive: How the Credit Score is Calculated

The platform uses a **Deterministic Credit Simulator** (`backend/src/services/creditBureau.js`). 

In a real environment, we would make an API call to CIBIL. Because this is a simulation, we need a way to generate a score that looks real and **never changes** for the same user when they refresh the page.

We achieve this using mathematical "Seeding":

1. **The Seed:** When a user asks for their credit score, we take their `user_id` (a UUID string).
2. **The Base Hash:** We convert every character of the UUID into its ASCII char code and sum them up. E.g., `a1b2-c3...` might sum up to `2450`.
3. **The Score Generation:** 
   - We take `(sum % 400) + 400`. This guarantees a mathematical baseline score between 400 and 800.
   - We then apply "Profile Modifiers". If the user's `employment_type` in the `users` table is `salaried`, we add +45 to their score. If their `monthly_income` is > ₹1,00,000, we add +30.
   - Finally, we bound the score. If the math pushes it above 900, we cap it at 900. If it falls below 300, we cap it at 300.
4. **Derived Metrics:** 
   - **Active Accounts:** Calculated using `(sum % 5) + 1` (Resulting in 1 to 5 accounts).
   - **Total Outstanding:** Calculated as `(sum % 10) * 50000` (Resulting in ₹0 to ₹4,50,000 in debt).
   - **Credit Utilization:** Calculated as `(outstanding / max_limit) * 100`.

**Why this is genius:** Because the UUID and Profile don't change randomly, `(sum % 400)` will ALWAYS yield the exact same math result. The user gets a completely personalized, static credit score that reacts logically to their income and employment data without relying on a real bureau!

## 4. Deep Dive: The Eligibility Engine Rules

Once the Credit Score is generated, the `EligibilityEngine` evaluates 7 core banking parameters. Each rule has a strict threshold:

1. **Age Check:** Must be 21-60 years old.
2. **Income Check:** Monthly income must be > ₹15,000.
3. **Credit Score Check:** Must be > 650.
4. **FOIR (Fixed Obligation to Income Ratio):** Calculates `(Existing EMIs + Proposed New EMI) / Monthly Income`. The total debt burden must not exceed 50% of their income.
5. **LTI (Loan to Income Ratio):** The total loan amount cannot be more than 10x their monthly salary.
6. **Employment Stability:** Must have > 1 year of employment.
7. **Amount Check:** The requested amount must fall within the allowed bounds for the specific `loan_type` (e.g., Personal loans are capped at 10L, Home loans at 1Cr).

If a user fails a critical rule (like Credit Score < 650), their `eligibility_score` drops significantly, and the system auto-marks them as `ineligible`, saving the Admin time!
